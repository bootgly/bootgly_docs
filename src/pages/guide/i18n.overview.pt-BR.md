# i18n

O Bootgly traz um contrato mínimo de i18n nativo e sem dependências em
`Bootgly\ABI\Data\Language`: `translate()` + catálogos de mensagens + negociação de locale.
As keys são **strings naturais da língua-fonte** — a mensagem na língua-fonte (inglês por
padrão) *é* a key do catálogo, e qualquer miss retorna a própria key. Com zero arquivos de
catálogo a aplicação continua funcionando em inglês; traduções são puramente aditivas.

Três superfícies do framework já consomem o contrato: **mensagens de validação**,
**templates** (a diretiva `@translate`) e **páginas de erro** (o Catcher de produção).

> [!NOTE]
> Este é o contrato congelado da v0.24: as assinaturas públicas já antecipam pluralização,
> domains e locale por chamada. Catálogos embutidos do framework vêm depois — projetos
> fornecem as próprias traduções.

## Começando

Um catálogo é um arquivo PHP simples em `{root}/{locale}/{domain}.php` retornando pares
`mensagem-fonte → tradução`:

```php
<?php
// catalogs/pt-BR/app.php

return [
   'Welcome, {name}!' => 'Bem-vindo, {name}!',
];
```

Registre o root, escolha o locale, traduza:

```php
use Bootgly\ABI\Data\Language;

Language::load(__DIR__ . '/catalogs');
Language::$locale = 'pt-BR';

echo Language::translate('Welcome, {name}!', ['name' => 'Bootgly']);
// Bem-vindo, Bootgly!
```

Em um **projeto** Bootgly, o diretório `catalogs/` é registrado automaticamente no boot —
basta criar `{project}/catalogs/pt-BR/app.php` e dispensar o `load()`.

Lookups fazem fallback progressivo: `pt-BR` → `pt` → a própria mensagem-fonte. Um arquivo
`catalogs/pt/errors.php` atende, portanto, requisições `pt-BR`, `pt-PT` e `pt` puro até
existir um catálogo mais específico.

## Placeholders

Placeholders `{token}` são substituídos depois do lookup no catálogo — os mesmos tokens
funcionam na mensagem-fonte e em cada tradução:

```php
Language::translate('{field} must be at least {limit}.', [
   'field' => 'age',
   'limit' => 18,
]);
// age must be at least 18.
```

Tokens sem substituição passam literais. `translate()` nunca escapa a saída — escape
pertence à fronteira de output (a diretiva `@translate` escapa; a bag de erros de validação
permanece texto puro).

## Plurais

Valores de catálogo (e mensagens-fonte) podem carregar formas separadas por `|`. Passar
`count:` seleciona uma — `count == 1` pega a primeira forma, o resto pega a última — e
preenche automaticamente o placeholder `{count}`:

```php
Language::translate('{count} result|{count} results', count: 7);
// 7 results

Language::translate('{count} result|{count} results', count: 1);
// 1 result
```

Sem `count:`, pipes são texto comum e inerte.

## Negociação de locale

`Language::negotiate()` casa preferências ordenadas com os locales que seus catálogos de
fato oferecem (os nomes dos diretórios `{locale}/`), com fallback de subtag e expansão
regional. A **língua-fonte é sempre a primeira offer implícita** — ela não precisa de
catálogo, então `['en', 'pt-BR']` escolhe `en` e o curinga `*` resolve para a fonte:

```php
Language::$locale = Language::negotiate(['pt-BR', 'en']); // dirs de catálogo: pt/ → 'pt'
Language::negotiate(['en', 'pt-BR']); // 'en' — a fonte vence preferências menores
Language::negotiate(['*']);           // 'en' — curinga resolve para a fonte
```

Diretórios de locale casam pela tag normalizada, então um diretório POSIX `pt_BR/`
negocia e carrega como `pt-BR`.

**Exclusões** do cliente são respeitadas: um range com `q=0` significa "não aceitável"
(RFC 9110) e remove as offers que ele cobre — mesmo através do curinga `*` — a menos que um
range aceito mais específico as reinclua. Quando toda offer é recusada, o header é
desconsiderado e a fonte é servida:

```php
Language::negotiate(['*'], excluded: ['en']); // 'pt' — o curinga desvia da fonte recusada
Language::negotiate([], excluded: ['*']);     // 'en' — tudo recusado: header desconsiderado
```

### Negociação automática por request (Web)

No servidor HTTP não há nada para configurar. Com catálogos registrados, cada request
negocia o `Accept-Language` automaticamente antes do roteamento — exclusões `q=0`
incluídas — e a atribuição incondicional também é o reset por request, então uma requisição
`pt-BR` nunca vaza o locale para a próxima. Apps sem catálogos pagam uma leitura estática
por request.

Com catálogos registrados, toda response também declara `Vary: Accept-Language`
automaticamente, para que CDNs e proxies compartilhados nunca reutilizem o corpo de uma
língua para outro cliente. O merge é token-aware (RFC 9110): um token `Accept-Language`
existente em qualquer caixa nunca duplica, um campo superstring como
`X-Accept-Language-Experiment` não satisfaz a checagem, e um `Vary: *` definido pelo
handler fica como está. O route-cache interno também é particionado pelo locale negociado.

```text
GET /error HTTP/1.1
Accept-Language: pt-BR
→ <html lang="pt-BR"> ... <p>Erro interno do servidor</p>
```

## Mensagens de validação localizadas

Toda mensagem padrão de validator passa pelo domain `validation`. Os padrões em inglês são
as keys — um catálogo pt-BR completo:

```php
<?php
// catalogs/pt-BR/validation.php

return [
   '{field} is invalid.' => '{field} é inválido.',
   '{field} is required.' => '{field} é obrigatório.',
   '{field} must be a boolean.' => '{field} deve ser um booleano.',
   '{field} must be a valid date.' => '{field} deve ser uma data válida.',
   '{field} must be a valid date in the format {format}.' => '{field} deve ser uma data válida no formato {format}.',
   '{field} must be a valid email address.' => '{field} deve ser um endereço de e-mail válido.',
   '{field} must be a valid URL.' => '{field} deve ser uma URL válida.',
   '{field} must be an integer.' => '{field} deve ser um número inteiro.',
   '{field} must be at least {limit}.' => '{field} deve ser no mínimo {limit}.',
   '{field} must be at most {limit}.' => '{field} deve ser no máximo {limit}.',
   '{field} must be at most {limit} bytes.' => '{field} deve ter no máximo {limit} bytes.',
   '{field} must be one of the allowed values.' => '{field} deve ser um dos valores permitidos.',
   '{field} must have an allowed extension.' => '{field} deve ter uma extensão permitida.',
   '{field} must have an allowed MIME type.' => '{field} deve ter um tipo MIME permitido.',
   '{field} confirmation does not match.' => 'A confirmação de {field} não confere.',
   '{field} has an invalid format.' => '{field} tem um formato inválido.',
];
```

```php
use Bootgly\ADI\Validation;
use Bootgly\ADI\Validators\Required;

Language::$locale = 'pt-BR';

$Validation = new Validation(
   source: [],
   rules: ['email' => [new Required]]
);

$Validation->errors['email'][0]; // email é obrigatório.
```

Overrides via `message:` também passam pelos catálogos (logo são keys traduzíveis) e ganham
suporte ao placeholder `{field}`; sem entrada no catálogo, renderizam literais, exatamente
como antes.

## Templates

A diretiva `@translate` é a forma canônica em templates — a saída é **escapada por padrão**
(traduções interpolam dados de usuário). Os argumentos passam literalmente para
`Language::translate()`, então argumentos nomeados funcionam:

```text
@translate 'Welcome, {name}!', ['name' => $name];

@translate '{count} result|{count} results', count: $total;

@translate 'Sign in', domain: 'auth';
```

Escape a própria diretiva com `@@translate`. Para traduções raw de HTML confiável, use a
diretiva de output raw: `@> \Bootgly\ABI\Data\Language::translate('key');`.

## Páginas de erro localizadas

Em Production/Staging, a clean page embutida e o corpo JSON de erro traduzem a reason phrase
HTTP pelo domain `errors` — a reason phrase é a key:

```php
<?php
// catalogs/pt-BR/errors.php

return [
   'Not Found' => 'Não encontrado',
   'Internal Server Error' => 'Erro interno do servidor',
];
```

O atributo `lang` do HTML segue o locale ativo. A **status line do wire permanece em inglês**
(`HTTP/1.1 500 Internal Server Error`) — só o texto do corpo é localizado, conforme a
semântica das RFCs.

Páginas totalmente customizadas continuam funcionando: uma view
`views/errors/{code}.template.php` é renderizada com o locale da requisição já ativo, então
`@translate` dentro dela simplesmente funciona.

## Console e scripts

Fora da plataforma Web, atribua o locale explicitamente. `Locales::normalize()` entende
POSIX, então o locale do ambiente está a uma linha de distância:

```php
use function getenv;
use Bootgly\ABI\Data\Language;

Language::load(__DIR__ . '/catalogs');
Language::$locale = Language::negotiate([(string) getenv('LANG')]); // pt_BR.UTF-8 → pt-BR
```

## Ressalvas

- **Catálogos cacheiam por worker.** Arquivos sofrem `require` uma vez por (root, locale,
  domain) e ficam pela vida do worker (amigável ao opcache). Editar um catálogo em servidor
  de longa duração exige reload/restart.
- **Respostas deferidas (Fibers) são locale-safe.** `$Response->defer()` captura o locale
  da requisição e o vincula ao Fiber do job — traduções dentro do trabalho deferido mantêm
  a língua da requisição que o agendou, mesmo com outras requisições intercaladas. Trabalho
  assíncrono customizado fora do `defer()` pode fazer o mesmo com `Language::bind()` /
  `Language::unbind()`.
- **Editar a língua-fonte orfana traduções.** Keys são o texto-fonte (trade-off do gettext):
  reescrever a mensagem em inglês renomeia a key — atualize os catálogos junto com o fonte.

## Referência

### `Bootgly\ABI\Data\Language`

```php
public static string $source = 'en';
```

A língua em que as strings-fonte (keys) são escritas. Lookups para ela fazem short-circuit
dos catálogos.

```php
public static string $locale = 'en';
```

O locale padrão (síncrono). Atribua diretamente; no servidor HTTP ele é renegociado (e
portanto resetado) a cada request quando há catálogos registrados. Um binding de Fiber tem
precedência sobre ele: dentro de um Fiber vinculado o locale efetivo é `resolve()`, não
esta property — use `bind()` para mudar intencionalmente o locale de um Fiber vinculado.

```php
public static array $roots = [];
```

Roots de catálogo registrados, em ordem de registro. Leia como probe de "i18n habilitado";
gerencie via `load()`.

```php
public static function load (string $root): void
```

Registra um diretório root de catálogos (`{root}/{locale}/{domain}.php`). Roots registrados
depois têm prioridade sobre os anteriores. Duplicatas são ignoradas.

```php
public static function translate (string $message, array $substitutions = [], null|int|float $count = null, string $domain = 'app', null|string $locale = null): string
```

Traduz uma mensagem-fonte para o locale ativo (ou informado). Nunca lança exceção: em
qualquer miss, a própria mensagem é o resultado, e uma substituição cujo cast `Stringable`
lança deixa o token intacto. `$count` seleciona uma forma plural separada por `|` (fallback
de duas formas, não CLDR) e preenche `{count}`; `$domain` escolhe o arquivo de catálogo
(`app` por padrão); `$locale` sobrepõe o locale ativo só nesta chamada. Nenhum escape de
saída é feito.

```php
public static function negotiate (array $preferred, array $excluded = []): string
```

Escolhe o melhor locale disponível para as preferências ordenadas — e.g.
`$Request->languages`. A língua-fonte é a primeira offer implícita, seguida dos diretórios
de locale dos roots registrados (normalizados). `$excluded` carrega os ranges recusados
pelo cliente (`q=0`, e.g. `$Request->exclusions`): offers recusadas são puladas mesmo
através do curinga `*`, e quando tudo é recusado o header é desconsiderado (a fonte é
servida). Pura: retorna a escolha (caindo para `Language::$source`) e nunca muta estado.

```php
public static function bind (string $locale): void
```

Vincula um locale ao Fiber corrente — trabalho deferido/assíncrono traduz sob o locale da
requisição que o agendou, imune a requisições intercaladas reatribuindo
`Language::$locale`. `$Response->defer()` chama isso automaticamente. No-op fora de um
Fiber.

```php
public static function unbind (): void
```

Remove o vínculo de locale do Fiber corrente (job terminou / Fiber devolvido ao pool).
Automático para jobs de `defer()`.

```php
public static function resolve (): string
```

Resolve o locale efetivo: o vínculo do Fiber corrente quando existe, senão
`Language::$locale`.

```php
public static function reset (): void
```

Restaura o estado de boot: descarta roots e caches; `$source`/`$locale` voltam a `en`.
Auxiliar de ciclo de vida de testes.

### `Bootgly\ABI\Data\Language\Locales`

```php
public static function normalize (string $locale): string
```

Normaliza uma tag BCP 47 ou POSIX para a forma canônica (`pt_BR.UTF-8` → `pt-BR`,
`PT-br` → `pt-BR`). `C`, `POSIX`, vazio ou malformado normaliza para `''`.

```php
public static function chain (string $locale): array
```

Expande uma tag normalizada na cadeia de lookup por truncamento progressivo:
`pt-BR` → `['pt-BR', 'pt']`.

```php
public static function choose (array $preferred, array $available, array $excluded = []): null|string
```

Lookup RFC 4647 (lite): por preferência — match exato, truncamento (`pt-BR` casa `pt`),
expansão regional (`pt` casa o primeiro `pt-*`), curinga `*`. A ordem do cliente vence.
Ranges em `$excluded` (`q=0`) removem as offers que cobrem antes do matching, a menos que
um range preferido mais específico reinclua a offer (RFC 9110 §12.5.4). Retorna a entrada
disponível como fornecida, ou `null`.

### Diretiva `@translate`

```text
@translate <argumentos>;
```

Compila para um echo de `Language::translate(<argumentos>)` escapado em HTML
(`htmlspecialchars` com `ENT_QUOTES | ENT_SUBSTITUTE`, UTF-8). A lista de argumentos passa
literal — argumentos posicionais e nomeados (`count:`, `domain:`, `locale:`) funcionam.
`@@translate` emite o texto literal da diretiva.
